# Brain-repo eval harness

[Promptfoo](https://promptfoo.dev) is the eval framework per
`AGENT_PRODUCT_CYCLE.md` §13. Goldens live under `/evals/<agent-id>/golden/`
(planned, currently inline in `tests.yaml`); rubrics under `/evals/rubrics/`
are scored by an LLM-as-judge running on `claude-opus-4-7` (`MODEL_EVAL`).

## Layout

```
/evals/
  package.json                  ← Promptfoo dep + npm scripts
  lib/build-prompt.js           ← Loads live agent prompt + builds chat msgs
  judge-prompts/
    rubric-judge.txt            ← The grading prompt template (used by all rubrics)
  rubrics/
    ceo-orchestration-rubric.md
    manager-decomposition-rubric.md
    manager-review-rubric.md
    worker-output-rubric.md
  framework-config/             ← Promptfoo global notes (see ../README in this dir)
  ceo/
    promptfooconfig.yaml        ← Agent runtime + judge wiring
    tests.yaml                  ← 50 goldens
    results.json                ← Generated by `npm run eval:ceo`
  base-manager/{config,tests,results}
  base-worker/{config,tests,results}
  adversarial/                  ← Phase 13 deliverable, TODO
```

## Prerequisites

- Node.js ≥ 18 (Promptfoo CLI requires it).
- `ANTHROPIC_API_KEY` exported in the shell. Promptfoo's Anthropic provider
  reads this directly. Source from Doppler in production environments.
- Models pinned in each `promptfooconfig.yaml`:
  - **Agent runtime:** `claude-sonnet-4-6` (CEO + base-manager + base-worker)
  - **Judge:** `claude-opus-4-7`
  - These match the `MODEL_*` env vars in `.env.example`. When upgrading
    pin the date-stamped string per §Model Selection Strategy.

## Run

```bash
cd evals
npm install                     # one-time; pins promptfoo
export ANTHROPIC_API_KEY=...    # via Doppler in prod
npm run eval:ceo                # CEO suite
npm run eval:base-manager       # base-manager suite
npm run eval:base-worker        # base-worker suite
npm run eval:all                # full sweep
npm run view                    # open the Promptfoo UI on the latest run
```

Each suite writes a per-agent `results.json` with per-test scores, judge
rationales, and aggregate pass rates. Check it in only as a baseline (see
"Baseline workflow" below).

## Cost expectation per full sweep

Rough order-of-magnitude estimate, current Anthropic API pricing:

| Step | Tokens (in/out) | Per-call | × 150 |
|------|-----------------|----------|-------|
| Agent runtime (sonnet-4-6)   | ~3K / ~1.5K | ~$0.03 | ~$4.50 |
| LLM-as-judge (opus-4-7)      | ~5K / ~0.5K | ~$0.11 | ~$16.50 |
| **Total per full sweep**     |             |        | **~$20** |

Re-run to capture noise floor: §13 says run the suite 3× on an unchanged
prompt to measure variance. Plan ~3× the above for a noise calibration.

## Baseline workflow (Phase 06 PR-anti-drift gate)

Before any prompt change ships:

1. Run the suite on `master` (current production prompts) → record `before`
   scores per test in `<agent>/baseline.json`.
2. Switch to the change branch, re-run → `after` scores.
3. Diff: any per-dimension regression > 2× the noise floor blocks merge per
   §13 / §Phase 06. Smaller regressions need a written explanation in the PR.
4. Paste before/after summary into the PR body (the
   `.github/PULL_REQUEST_TEMPLATE.md` "Eval run completed" line).

## Adding a golden

1. Decide which agent + which use case (decomposition? review? decision-tree?).
2. Append to `<agent>/tests.yaml` with fields:
   ```yaml
   - description: "<agent>-<NNN>: short title"
     vars:
       rubric: file://../rubrics/<rubric>.md   # if not the default
       user_input: |
         <prompt to send>
       expected_behavior: |
         <what the judge should look for>
   ```
3. Re-run the suite locally to confirm the test executes (judge produces
   valid JSON, no schema errors).
4. Append a one-line entry to `<agent>/CHANGELOG.md` (append-only per §Git &
   Versioning > Eval golden datasets).
5. Open a PR. Do **not** copy the golden into any agent's system prompt or
   `<examples>` block — contamination rule per §Prompt Engineering Standards.

## Rubric mapping

| Agent             | Use case                | Rubric                                  |
|-------------------|-------------------------|------------------------------------------|
| ceo               | All cases               | ceo-orchestration-rubric.md              |
| base-manager      | Decomposition (01-15)   | manager-decomposition-rubric.md          |
| base-manager      | Review/coaching/health  | manager-review-rubric.md                 |
| base-worker       | All cases               | worker-output-rubric.md                  |

The `vars.rubric` field on each test selects the rubric. The judge grades
strictly per rubric; passing requires every dimension ≥ 3 *and* mean ≥ 4
per `judge-prompts/rubric-judge.txt`.

## Known limitations of the current baseline

- The CEO and base-manager/worker prompts are **STUB / TEMPLATE** — they
  carry "TODO at hire" markers (no domain examples, generic tool scope).
  Baseline scores will reflect that. The point is to capture *where the
  prompts are now* so future edits can be compared, not to pass the gate
  on day one.
- Tests inject synthetic manager/worker rosters into `user_input` because
  `/registry/agents.json` does not yet contain concrete domain agents
  (Milestone 5 introduces those). Replace the synthetic rosters with
  registry references once concrete agents land.
- `adversarial/` red-team prompts are a Phase 13 / Milestone 3 deliverable,
  not part of this baseline.
